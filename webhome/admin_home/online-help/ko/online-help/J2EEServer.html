엔진 컨테이너

네비게이션 경로: Home > node-name > 엔진 컨테이너

개요

이 페이지에서는 이 노드에 설정되어 있는 엔진 컨테이너의 목록을 표의 형태로 볼 수 있다. 표에는 컨테이너 별로 다음 정보가 표시된다.

이름

엔진 컨테이너의 이름. 클릭하면 해당 컨테이너의 엔진 컨테이너 설정 페이지로 이동한다.

상태

이 컨테이너의 상태에 따라 Running, Stopped 중 한 가지 값이 나타난다. 상태에 따라 아래에서 설명하는 명령 버튼들이 활성/비활성화 된다.

명령

시작버튼을 누르면 컨테이너를 시작시켜 Running 상태가 되고 다운버튼을 누르면 컨테이너를 다운시켜 Stopped 상태가 된다. Running 상태의 컨테이너는 다운 버튼만, Stopped 상태의 컨테이너는 시작 버튼만 활성화 되어 있어 현 상태에서 할 수 있는 명령을 알려준다.

엔진 컨테이너 복사

복사 아이콘( )을 클릭하면 엔진 컨테이너의 복사 로 이동한다. 이때는 새로운 엔진 컨테이너를 생성하는 것과는 달리 엔진의 종류를 추가 제거 할 수 없다.

삭제

Stopped 상태의 엔진 컨테이너의 경우만 휴지통 아이콘( )이 보인다. 이를 클릭하면 삭제 확인 대화상자가 나타나고 확인 버튼을 클릭하면 엔진 컨테이너를 노드에서 제거한다.

또한 표 아래의 버튼으로 엔진 컨테이너를 제어 할 수 있다.

새 엔진 컨테이너 생성

엔진 컨테이너 표 아래에 이 버튼을 클릭하면 엔진 컨테이너 생성 페이지로 이동한다. 자세한 내용은 엔진 컨테이너 생성 페이지의 도움말을 참조하고 생성을 하고 나면 엔진 컨테이너 목록에 새로운 엔진 컨테이너가 Stopped 상태로 추가된다.

모두 시작

엔진 컨테이너 표 아래에 이 버튼을 클릭하면 여러 컨테이너 중 Stopped되어 있는 모든 컨테이너를 시작 시킨다.

모두 다운

엔진 컨테이너 표 아래에 이 버튼을 클릭하면 여러 컨테이너 중 Running되어 있는 모든 컨테이너를 시작 시킨다.

엔진 컨테이너 생성

새로운 엔진 컨테이너의 생성

새 엔진 컨테이너를 생성하기 위해서는 이 페이지에서는 이름과 엔진의 기본 사항 등 핵심 요소만을 설정한다. 설정을 하고 생성 버튼을 누르면 생성 확인 대화 상자가 뜨고 확인을 클릭하면 엔진 컨테이너 목록에 새로운 엔진 컨테이너가 Stopped 상태로 추가된다. 여기서 해당 엔진 컨테이너를 클릭하면 엔진 컨테이너 설정 페이지에서 자세한 설정을 할 수 있다.

엔진 컨테이너의 복사

이 페이지에 나오는 요소들은 모두 엔진 컨테이너마다 달라야 하는 요소들이다. *로 표시로 되어 있는 필수 요소인 엔진 컨테이너 이름과 각각의 엔진 이름은 내부적으로 현 컨테이너와 다른 이름이 적용되어 있어서 특별한 이름을 사용하고자 하는 경우가 아니면 그대로 사용해도 무방하다. 그러나 선택 요소인 id와 base port는 현 컨테이너와 같은 설정이 적용되어 있으므로 이 요소를 사용하고 싶을 때에는 수정하고 생성 버튼을 눌러야 id와 port가 충돌하지 않는다. 생성 버튼을 클릭하면 각종 포트를 변경하라는 메시지와 함께 복사 확인 대화 상자가 뜬다. 대화 상자를 확인을 클릭하고 나면 엔진 컨테이너 목록에 새로운 엔진 컨테이너가 Stopped 상태로 추가된다.

엔진 컨테이너 설정

이 페이지에서는 특정 엔진 컨테이너의 세부적인 설정을 엔진 컨테이너 설정 - 기본 설정, 엔진 컨테이너 설정 - 에러 로그, 엔진 컨테이너 설정 - 사용자 로그, 엔진 컨테이너 설정 - 상호 운용, 엔진 컨테이너 설정 - 라이프 사이클, 엔진 컨테이너 설정 - 리소스 참조, 엔진 컨테이너 설정 - 기타 서브탭을 통해 확인 및 수정할 수 있다. 그러나 Running 상태의 엔진 컨테이너를 여기서 설정을 바꾼다고 해도 재 시작할 때까지 적용되지 않음을 주의한다.

엔진 컨테이너 설정 - 기본 설정

엔진 컨테이너의 가장 기본적인 것들에 대한 설정을 한다.

Name

JEUS 엔진 컨테이너의 식별자로 매니저 내에서 유일한 이름을 가진다. 엔진 컨테이너의 이름은 사용자가 설정할 수 있다. 만약 이름이 “default”라고 설정되면 JEUS 매니저와 같은 JVM인 Default 엔진 컨테이너로 구동된다. 주의할 것은 노드의 이름은 영문자와 숫자의 조합으로 이루어지되, 영문자로 시작하는 문자열이어야 한다.

고급 선택 사항
Id

엔진 컨테이너의 Id로 0부터 128까지의 값을 갖는다. 이 값은 사용자가 설정할 수 있으며 기본 값은 엔진 컨테이너 이름의 hashcode를 바탕으로 만든다. 다른 엔진 컨테이너와 Id가 겹치면 엔진 컨테이너 부팅이 실패한다.

Base Port

엔진 컨테이너의 listen 포트이다. 이 값은 사용자가 설정할 수 있으며 기본적으로 base port + 15 + containerID * 10 의 값이 container의 base port로 사용된다.

Command Option

엔진 컨테이너를 실행하기 위해 개별적인 JVM에 추가할 파라미터들을 선언하는데 사용된다. 표준 JVM Property 뿐 아니라 JEUS에서 지정한 System Property도 설정할 수 있다.

Jvm Option

엔진 컨테이너를 실행하기 위해 개별적인 JVM에 추가할 파라미터들을 선언하는데 사용된다. 하나의 항목에 하나의 옵션을 설정할 수 있다. 하위호환성을 위해 Command Option에 설정된 옵션이 먼저 적용된다. Command Option설정보다 Jvm Option설정을 권장한다.

User Class Path

엔진 컨테이너에 system classpath를 추가한다. 이 설정은 디폴트(default) 엔진 컨테이너에는 적용되지 않는다. 디폴트 엔진 컨테이너에 적용하기 위해서는 jeus.server.classpath system property를 사용하여야 한다. 디폴트 컨테이너가 아닌 컨테이너에 대해서는 JEUSMain.xml의 설정이 system property에 우선한다. 구분자는 ";" (유닉스의 경우 “:”)을 사용한다.

Log Stdout To Raw Format

엔진 컨테이너 로그에 출력되는 표준출력/에러의 출력방식에 대해 설정한다. 기본적으로 엔진 컨테이너에 한 설정이 최우선시 되고 이 설정이 없다면 매니저의 설정을 따른다. 엔진 컨테이너와 매니저 모두에 이 설정이 없다면 기본 값은 true이다.

엔진 컨테이너 설정 - 에러 로그

엔진 컨테이너에서 사용할 로거를 설정한다.

이 페이지에서는 JEUS 엔진 컨테이너에 설정된 로거의 목록을 보여주고, 등록된 로거의 설정을 변경/삭제하거나 새로운 로거를 추가할 수 있다.

로거 테이블

이 테이블에서는 등록된 로거의 이름과 Level을 볼 수 있으며, 등록된 로거를 삭제할 수도 있다.

이름

등록된 로거의 이름을 나타내며, 이 로거의 링크를 클릭하면 해당 로거의 설정을 변경할 수 있는 페이지(로깅 - 설정)로 이동한다.

LEVEL

로거의 로깅 레벨을 나타낸다. 로깅 레벨은 아래와 같이 12개가 있다.

레 벨설 명
FATALJEUS 4.x대의 logger의 level로 compatibility를 위해 지원한다.SEVERE에 해당하는 level이다.
NOTICEWARNING에 해당하는 level이다.
INFORMATIONINFO에 해당하는 level이다.
DEBUGFINE에 해당하는 level이다.
SEVERE치명적인 실패, 오류(serious failure)에 대한 로깅 레벨
WARNING잠재적인 문제(potential problem)에 대한 로깅 레벨
INFO안내 메세지(information message)에 대한 로깅 레벨
CONFIG설정 메세지(static configuration message)에 대한 로깅 레벨

표의 맨 마지막 컬럼 나타난 을 누르면 해당 로거를 삭제할 것인지 확인 메세지가 출력되고, '확인'을 누르면 삭제된다.

로거 추가

표 아래에 ' 로거 추가'를 클릭하면 로깅 - 로거 추가 로 이동하여 새로운 로거를 등록할 수 있다.

고급 선택 사항
Name

등록된 로거의 이름을 나타내며, 이 로거의 링크를 클릭하면 해당 로거의 설정을 변경할 수 있는 페이지(로깅 - 설정)로 이동한다.

Use Parent Handlers

이 로거에서 상위 로거(parent logger)의 핸들러들을 사용할지를 설정한다. 기본 값은 true로 상위의 handler를 사용한다. 하지만 "jeus" 로거의 경우는 JEUS의 최상위 로거이기 때문에 시스템 내부적으로 무조건 false로 동작한다.

Filter Class

이 로거에서 사용할 필터 클래스의 이름을 fully qualified class name으로 설정한다.

로거 핸들러

등록한 로거에서 사용할 핸들러를 설정한다.

추가하고 싶은 핸들러를 클릭하면 새로운 핸들러를 추가할 수 있는 페이지(새 로그 핸들러 생성)로 이동한다.

로그 핸들러 테이블

핸들러 테이블에 대한 설명은 로그 핸들러 페이지를 참고한다.

엔진 컨테이너 설정 - 상호 운용

RMI/IIOP 상호 운영성(interoperability)에 대해서 설정한다. 이 설정이 true이면 상호 운영을 enable 한다. 기본값은 false 이다. EJB RMI/IIOP를 사용한다면 반드시 설정되어야 한다.

상호 운용 SSL 구성

keystore path

현 도메인에 적용할 keystore file경로를 정의함.

keystore alias

keystore 파일의 keyEntry 타입의 인증서가 여러개인 경우 명시적으로 alias값으로 해당 서버 인증에 필요한 인증서를 가리키도록 함.

keystore password

현 도메인에 적용할 keystore file에 대한 password를 정의함. password를 암호화해서 저장할 때에는 {algorithm}ciphertext 와 같은 형식으로 쓴다. ex) {DES}FQrLbQ/D8O1lDVS71L28rw==

keystore keypassword

현 도메인에 적용할 keystore file 안의 key들에 대한 keypassword를 정의함. password를 암호화해서 저장할 때에는 {algorithm}ciphertext 와 같은 형식으로 쓴다. ex) {DES}FQrLbQ/D8O1lDVS71L28rw==

truststore path

현 도메인에 적용할 truststore file 경로를 정의함.

truststore password

현 도메인에 적용할 truststore file에 대한 password를 정의함. password를 암호화해서 저장할 때에는 {algorithm}ciphertext 와 같은 형식으로 쓴다. ex) {DES}FQrLbQ/D8O1lDVS71L28rw==

엔진 컨테이너 설정 - 라이프 사이클

엔진 컨테이너의 각종 lifecylcle 이벤트에 호출하고 싶은 method 지정할 수 있다.

클래스 이름

라이프 사이클 이벤트의 콜백 메소드가 존재하는 이름을 선택한다.

메소드 이름

라이프 사이클 이벤트의 콜백 메소드 이름을 선택한다. 메소드를 선택하고 삭제 를 누르면 해당 설정이 삭제된다.

새 메소드 추가

Class Name

라이프 사이클 이벤트의 콜백 메소드가 존재하는 클래스 이름을 설정한다.

Method Name

라이프 사이클 이벤트를 받으면 수행할 콜백 메소드 이름을 설정한다.

인자

라이프 사이클 이벤트의 콜백 메소드의 인자값을 설정한다. "타입=값" 형식으로 설정한다.

Invocation Type

설정한 메소드가 호출될 시점을 선택한다.

BOOT, BEFORE_DEPLOY, AFTER_DEPLOY, READY, BEFORE_UNDEPLOY, AFTER_UNDEPLOY가 있다.

Library Ref

application에서 사용할 shared library를 정보를 설정한다.

Library Name

shared library 이름을 설정한다.

Fail On Error

해당 shared library를 찾지 못할 경우 디플로이가 실패되어야 할 지 여부를 설정한다. 기본값은 false로 디플로이를 성공하도록 한다.

Specification Version

사용할 shared library 스펙(specification) 버전을 지정한다.

Exact Match

정확히 매치되는 버전이 필요한지 여부를 설정한다. 기본값은 false이다.

Version

정확히 매치되는 버전을 적는다.

Implementation Version

사용할 shared library 구현(implementation) 버전을 지정한다.

Exact Match

정확히 매치되는 버전이 필요한지 여부를 설정한다. 기본값은 false이다.

Version

정확히 매치되는 버전을 적는다.

엔진 컨테이너 설정 - 리소스 참조

이 엔진 컨테이너에 등록 되어 있는 리소스 참조 목록을 표의 형태로 볼 수 있다.

참조 이름

소스 코드상에서 사용하는 리소스의 참조 이름. 표에 나타나는 참조이름을 클릭하면 리소스 참조에 대해 수정할 수 있는 페이지로 이동한다.

익스포트 이름

네이밍 서버에 등록된 리소스 이름. JNDI 참조를 통해 리소스를 얻을 때 클라이언트는 이 익스포트 이름을 사용한다. 위의 이름과는 다를 수 있다.

삭제

휴지통 아이콘( )을 클릭하면 삭제 확인 대화상자가 나타나고 확인 버튼을 누르면 리소스 참조를 이 엔진 컨테이너에서 제거한다.

표 아래의 추가 버튼을 누르면 새로운 리소스 참조를 등록 할 수 있다.

엔진 컨테이너 설정 - 기타

Start On Boot

부팅 시에 이 엔진 컨테이너를 띄울지를 정한다. 기본값은 true이다. 이 값이 false이면 부팅 시에 이 컨테이너를 띄우지 않는다.

Sequential Start

엔진 컨테이너들이 순차적으로 시작될 지 여부를 결정하는 설정이다. 만약 false로 설정될 경우 각 엔진 컨테이너가 동시에 시작된다. 기본값은 false 이다.

Auto Restart

해당 엔진 컨테이너가 비정상 종료 되었을 때 매니저에서 컨테이너를 자동으로 재기동 해 줄지 여부를 결정한다. 각 엔진 컨테이너마다 자신의 설정이 우선시 되고 엔진 컨테이너에 해당 설정이 없으면 매니저의 설정을 따른다. 엔진 컨테이너와 매니저 모두 이 설정이 없다면 기본 값은 true로 매니저에서 해당 컨테이너를 재기동 해준다.

Invocation Manager Action

엔진 컨테이너에서 Servlet/JSP, EJB Stateless 세션빈, 그리고 MDB와 같은 Stateless 메소드를 호출 하는 동안 사용하는 외부 리소스(external resource)인 JDBC connection과 Webt connection을 추적하여 connection이 닫히지 않은 경우 모드에 따라 적절한 처리를 해준다. 기본 동작은 Warning으로 warning message의 로그를 남긴다. NoAction을 선택하면 이 기능을 사용하지 않는것 이고 AutoClose를 선택하면 제대로 닫히지 않은 자원은 자동적으로 닫힌다.

Use MEJB

J2EE Management Spec에서 제시하는 MEJB를 사용할 것인지를 설정한다. 사용하지 않는다면 MEJB를 디플로이하지 않는다. 기본값은 false이다.

Application Path

어플리케이션 archive file들을 넣을 디렉토리를 지정한다. 상대경로인 경우에는 JEUS_HOME path에서의 상대경로이다. 어플리케이션은 이 element의 순서대로 검색된다. 기본값은 webhome/app_home이고 jeus.apphome system property을 설정했다면 이값이 기본값이 된다.

통계

JEUS 버전 정보와 JVM 힙 크기와 동작 시간 등을 보여준다. 새로고침 버튼을 클릭하면 실시간으로 변하는 것을 확인 할 수 있다.
