기본 설정

JMS 엔진의 이름과 Broker Name을 확인할 수 있다. Broker Name은 JMS 엔진들의 클러스터를 구성할 때 클러스터 내에서 각 엔진을 구분하기 위한 용도로 사용한다.

JMS 엔진의 이름은 엔진 추가 시에 결정되므로 변경할 수 있지만, Broker Name 설정은 변경 가능하다. Broker Name 필드 값을 입력하지 않으면 기본값으로 호스트 이름이 사용된다.

서비스 채널

하나의 JMS 엔진에는 네트워크 설정 등을 다르게 지정한 다수의 서비스 채널을 추가할 수 있다. 이 서비스 채널들은 클라이언트나 서버와 통신하는 채널 역할을 한다. 이 페이지에서는 간단하게 각 서비스 채널의 이름과 URL 정보를 표로 보여준다.

이름

서비스 채널의 이름. 클릭하면 해당 서비스 채널의 설정을 변경할 수 있는 서비스 채널 - 설정 페이지로 이동한다.

서비스 URL

해당 채널이 서비스를 제공하는 URL.

삭제

삭제 명령. 휴지통 아이콘()을 클릭하면 삭제 확인 대화 상자가 나타나고, 확인 버튼을 누르면 JMS 엔진에서 해당 서비스 채널을 삭제할 수 있다.

표 오른쪽 하단의 새 서비스 채널 생성 링크를 클릭하면 JMS 엔진에 추가할 새로운 서비스 채널을 설정하는 새 서비스 채널 생성 - 기본 설정 페이지로 이동한다.

에러 로그

JMS 엔진의 에러 로거를 설정하는 페이지이다. 로거 설정 및 로그 핸들러의 추가, 설정에 대해서는 로깅 - 설정의 설명을 참고하기 바란다.

액세스 로그

JMS 엔진의 액세스 로거를 설정하는 페이지이다. 로거 설정 및 로그 핸들러의 추가, 설정에 대해서는 로깅 - 설정의 설명을 참고하기 바란다.

Fail Over

JMS 엔진의 Fail Over기능을 설정하는 페이지이다.

  • Active Server

    이 JMS 엔진을 Active server로 사용하기 위해서 아래 값들을 설정한다.

    Listen Transport URL

    이 Active server가 접속을 대기할 URL을 설정한다. 기본적으로 gms라고 설정하면 된다.

    Check Health Response Timeout

    이 Active Server와 Standby Server사이의 Health check시의 Timeout을 설정한다.

  • Standby Server

    이 JMS 엔진을 Standby Server로 사용하기 위해서 아래 값들을 설정한다.

    Active Transport URL

    이 Standby Server가 접속할 Active Server의 URL을 설정한다. gms://<Active Server 브로커 이름> 과 같이 설정하면 된다.

    Check Health Response Timeout

    이 Standby Server와 Active Server사이의 Health Check시의 Timeout을 설정한다.

    Discovery Response Timeout

    이 Standby Server가 Active Server를 발견할 때 까지의 Timeout을 설정한다..

    Start Up Max Try Count

    이 Standby Server가 최초로 기동할 때 Active Server가 Failure상태라면 최종적으로 다시 확인해 볼 횟수를 설정한다.

    Fail Over Max Try Count

    이 Standby Server의 Active Server가 Failure상태라고 판단되었을 때 최종적으로 다시 확인해 볼 횟수를 설정한다.

    Auto Switchover

    이 Standby Server가 Fail Over하여 서비스중일 때, Active Server가 재기동되면 자동으로 Switchover를 할지 여부를 설정한다.

퍼지스턴스 스토어

여기에서는 JMS 엔진이 메시지 전달 보장 및 트랜잭션 로깅을 위해 사용하는 퍼지스턴스 스토어를 설정한다.

  • JDBC 퍼지스턴스 스토어

    JDBC 퍼지스턴스 스토어를 사용하기 위해선 아래 값들을 설정해야한다.

    Data Source

    JDBC 퍼지스턴스 스토어로 사용할 데이터소스를 선택한다. 사용 가능한 JDBC 데이터소스 항목 중에서 선택할 수 있다.

    Destination Table Name

    데스티네이션 정보를 저장할 테이블 이름. 지정하지 않으면 <BROKER-NAME>_DEST이 사용된다.

    Durable Subscriber Table Name

    Durable Subscriber 정보를 저장할 테이블 이름. 지정하지 않으면 <BROKER-NAME>_DSUB가 사용된다.

    Message Table Name

    메시지 정보를 저장할 테이블 이름. 지정하지 않으면 <BROKER-NAME>_MESGE가 사용된다.

    Subscription Message Table Name

    연결되어 있지 않은 Durable Subscriber에게 추후에 메시지를 전달하기 위해 메시지 정보를 저장할 테이블 이름. 지정하지 않으면 <BROKER-NAME>_DMSG가 사용된다.

    Transaction Table Name

    트랜잭션 상태들을 저장하는 테이블 이름. 지정하지 않으면 <BROKER-NAME>_TRAN이 사용된다.

  • 저널 퍼지스턴스 스토어

    JEUS 저널 스토어를 퍼지스턴스 스토어로 사용한다. JEUS 퍼지스턴스 스토어를 사용하기 위해선 아래 값들을 설정해야 한다.

    Destination Table Name

    데스티네이션 정보를 저장할 테이블 이름. 지정하지 않으면 <BROKER-NAME>_DEST이 사용된다.

    Durable Subscriber Table Name

    Durable Subscriber 정보를 저장할 테이블 이름. 지정하지 않으면 <BROKER-NAME>_DSUB가 사용된다.

    Base Dir

    저널 스토어를 생성할 디렉토리를 설정한다. 이 값이 설정되지 않는 경우 기본 디렉토리는 <JEUS_HOME>/logs/store/<BROKER_NAME>가 된다.

    Initial Log File Count

    저널 스토어를 처음 생성할 때 미리 생성할 로그파일의 갯수를 지정한다. 기본값은 2이다.

    Max Log File Count

    최대로 생성될 수 있는 로그파일의 갯수를 지정한다. 기본값은 20이다.

    Log File Size

    각 저널 로그파일의 크기를 지정한다. 기본값은 128mega bytes이다.

    Property

    위의 기본설정외에 고급 옵션들을 설정할 경우 사용하는데 자세한 내용은 JEUS MQ 안내서를 참조하기 바란다.

쓰레드 풀

이 페이지에서는 JMS 엔진이 내부적으로 사용하는 쓰레드 풀을 설정한다. JMS 엔진 내에서 동시에 수행 가능한 작업의 수에 영향을 미치는 부분이므로 동시 접속 가능한 클라이언트 수와 서버 부하 등을 고려한 튜닝이 필요한 부분이다.

Max

쓰레드 풀의 최대 크기. 반드시 값을 지정해야 한다.

Min

쓰레드 풀의 최소 크기. 기본값은 1이다.

Keep Alive Time

Min에서 설정한 개수를 초과하는 쓰레드 중 여기에서 지정한 시간동안 사용되지 않은 쓰레드는 소멸된다. 밀리초 단위로 설정하며 기본값은 300000(5분)이다.

클러스터

JMS 엔진의 클러스터링에 관한 설정을 관리한다.

Failure Detection Retries

클러스터 레벨에서 Failure를 판단할 때 최종적으로 재확인해 볼 횟수를 설정한다. 특별한 이유가 없는 이상 기본값을 사용한다.

Failure Detection Timeout

클러스터 레벨에서 Failure를 판단할 때의 Timeout을 설정한다. 특별한 이유가 없는 이상 기본값을 사용한다.

Failure Detection TCP Port

클러스터 레벨에서 Failure를 판단할 때 최종적으로 연결을 해볼 TCP소켓의 Port를 설정한다. 특별한 이유가 없는 이상 기본값을 사용한다.

Failure Detection TCP Timeout

클러스터 레벨에서 Failure를 판단할 때 최종적으로 TCP소켓으로 연결을 해볼 때의 Timeout을 설정한다. 특별한 이유가 없는 이상 기본값을 사용한다.

Failure Verification Timeout

클러스터 레벨에서 Failure를 판단할 때 최종적으로 TCP소켓으로 재연결을 해볼 때 재시도하는 주기를 설정한다. 특별한 이유가 없는 이상 기본값을 사용한다.

Discovery Timeout

클러스터 레벨에서 그룹의 리더를 정할 때 기다리는 최대의 시간을 설정한다. 이 시간이 지나도록 현재의 리더가 발견되지 않으면 자신이 리더가 된다. 특별한 이유가 없는 이상 기본값을 사용한다.

Broker Name

클러스터에 참여하는 모든 브로커의 이름을 추가한다.

Durable Subscriber

JMS 엔진을 시작할 때 등록할 Durable Subscriber의 목록을 관리한다. 설정되어 있는 Durable Subscriber의 표에서는 다음 정보를 확인할 수 있다.

클라이언트 아이디

Durable Subscriber의 Client Id.

이름

Durable Subscriber의 Name.

데스티네이션 이름

Durable Subscriber의 Destination Name.

메시지 선택자

Durable Subscriber의 Message Selector.

메시지 정렬

이 Durable Subscriber에 적용할 메시지 정렬 설정의 이름

삭제

휴지통 아이콘()을 클릭하면 삭제 확인 대화상자가 나타나고, 확인 버튼을 누르면 목록에서 해당 Durable Subscriber를 삭제할 수 있다.

목록 아래에는 새로 추가할 Durable Subscriber의 정보를 입력하기 위한 필드들이 있다.

Client Id

추가할 Durable Subscriber의 클라이언트 아이디를 입력한다.

Name

추가할 Durable Subscriber의 Subscription Name을 입력한다.

Destination Name

추가할 Durable Subscriber가 메시지를 수신하는 데스티네이션의 이름을 입력한다. JMS 데스티네이션 리소스에서 설정한 토픽 중 하나의 이름을 사용해야 한다.

Message Selector

새 Durable Subscriber의 메시지 셀렉터를 입력한다.

Durable Subscriber의 정보를 입력한 뒤 우측 하단의 추가 버튼을 누르면 목록에 새로운 Durable Subscriber가 나타난다. 새로 추가한 Durable Subscriber는 다음 JMS 엔진 기동시에 자동으로 등록되며, JEUS 노드 트리 상의 해당 토픽 아래에 나타난다.

Message Sort

Queue나 Durable Subscriber에 쌓이는 메시지들이 정렬될 수 있도록 하는 설정을 추가한다. 이곳에 기술해 둔 설정의 이름을 Queue나 Durable Subscriber에서 사용하게 된다.

Name

이 메시지 정렬 설정의 이름. Queue나 Durable Subscriber 설정에서 참조해서 사용하게 된다.

Key

메시지를 정렬할 때 기준이 되는 Key값을 설정한다. JMS 스펙에 명시된 기본 Property혹은 User Property의 Key값을 설정하면 된다.

Key Type

위에 설정한 Key에 대응하는 Value값으 Type을 설정한다. Boolean, Byte, Float, Integer, Double, String중에서 설정할 수 있다.

Direction

오름차순 혹은 내림차순의 여부를 설정한다.

트랜잭션

트랜잭션 탭에서는 WebAdmin을 통해 현재 pending상태인 트랜잭션들을 모니터링 하고 관리할 수 있는 기능을 제공한다.

  • pending 트랜잭션 모니터링

  • commit

  • rollback

pending 트랜잭션 모니터링

분산 트랜잭션에 참여 중인 JMS XAResource가 Two Phase Commit Protocol을 수행하던 중 'PREPARED'까지 진행된 상태에서 서버에 장애가 발생하면 pending 상태가 된다. 보통 pending 트랜잭션들은 트랜잭션 관리자에 의해서 다시 decision을 받고 commit이나 rollback을 자동으로 수행하게 되는데 트랜잭션 관리자에 어떤 문제가 생겨서 자동 복구가 되지 않는다면 시스템 관리자가 직접 decision을 내리고 commit이나 rollback을 수행해야한다.

이 페이지에서 보여주는 테이블의 각 컬럼은 아래의 의미를 가지고 있다.

ID

트랜잭션의 고유 아이디로서 내부적인 관리에 사용되는 아이디이다. 실제 Two Phase Commit에 사용되는 XID는 이 아이디에 마우스를 클릭하여 툴팁을 통해서 확인 할 수 있다.

상태

이 값은 트랜잭션의 현재 상태를 나타나는데 현재는 모두 'PREPARED'로 표시된다.

속성

이 컬럼에는 현재 이 트랜잭션에서 발생한 JMS 동작들의 정보를 보여준다. 이 동작에는 3가지가 있는데 아래와 같다.

  • Production : 메시지 생성하는 동작

  • Queue Consumption : Queue로 부터 메시지를 수신하고 ACK를 보내는 동작

  • Topic Durable Subscription : Topic의 Durable Subscriber가 메시지를 수신하고 ACK를 보내는 동작

명령

트랜잭션 관리자로 부터 어떤 decision을 받지 못하는 경우 시스템 관리자에 의해서 강제적인 commit이나 rollback을 수행할 수 있다.

commit

이 명령은 트랜잭션 관리자로 부터 어떤 decision을 받지 못한 트랜잭션을 시스템 관리자가 강제적으로 commit하는 명령이다. commit이 후 트랜잭션 관리자에 의해서 commit을 요청 받으면 아무런 예외없이 성공한 것으로 응답을 보내게 된다.

rollback

이 명령은 트랜잭션 관리자로 부터 어떤 decision을 받지 못한 트랜잭션을 시스템 관리자가 강제적으로 rollback하는 명령이다. rollback이 후 트랜잭션 관라자에 의해서 rollback을 요청 받으면 아무런 예외없이 성공한 것으로 응답을 보내게 된다.

배포

이 JMS 엔진의 설정을 JEUS 클러스터 내의 다른 JMS 엔진에 복사할 때 사용한다. 트리 구조의 JMS 엔진 목록에서 설정 배포 대상을 선택한 뒤 확인 버튼을 누르면 JMSMain.xml 설정이 복사된다. 설정이 덮어씌워진 JMS 엔진은 재시작 할 때 새로운 설정이 반영된다.
