개요

이 페이지에서는 리스너 쓰레드의 정보를 표 형태로 볼 수 있다.

1. 쓰레드 상태 정보 요약

각 리스너의 쓰레드 상태들을 요약한 정보를 보여준다.

활성중인 쓰레드

요청을 처리 중인 쓰레드의 수

대기 중인 쓰레드

요청을 받기 위해서 대기 중인 쓰레드의 수

블럭된 쓰레드

블럭된 쓰레드의 수. 블럭된 쓰레드의 정의는 "웹 리스너 - 쓰레드 풀" 페이지에서 설정한 Max Thread Active Time 값 이상의 시간 동안 active 상태인 쓰레드를 말한다.

재연결 중인 쓰레드

재연결 중인 쓰레드의 수

2. 표 설명

각 쓰레드의 정보를 요약한 표에 대한 설명을 한다.

쓰레드 아이디

각 쓰레드를 구분할 수 있는 쓰레드의 고유 이름

쓰레드 상태

현재 쓰레드의 상태 정보. waiting, active, blocked, reconnection 중 하나를 나타낸다.

경과 시간

쓰레드가 현재 상태로 지속된 시간

총 요청 수

현재 까지 해당 쓰레드가 처리한 요청 수

평균 처리 시간

각 요청에 대한 처리 시간의 합 / 총 요청 수

어플리케이션

현재 쓰레드가 처리 중인 요청의 URI를 나타낸다.

StackTrace

수행 중인 쓰레드의 stack trace 정보를 볼 때 사용한다. 수행중인 쓰레드의 쓰레드 아이디를 클릭하면 stack trace 정보를 확인 할 수 있다.

Interrupt

수행 중인 쓰레드에 interrupt를 시그널을 보낸다. EJB의 경우는 리모트 요청을 처리하는 RMI 쓰레드가 대상이 되고, Servlet의 경우는 Servlet 요청을 처리하는 쓰레드가 대상이 된다. 쓰레드에 interrupt가 걸리면 그 후에 요청되는 operation은 중단되고 InterruptedException이나 EJBException이 발생한다. 주로 요청을 처리하는 쓰레드가 예상한 시간보다 긴 시간동안 수행중인 상태로 있을 때 사용한다. 이 때 쓰레드의 stack trace는 달라 질 수 있고, interrupt status나 운영체제에 따라 interrupt에 걸리지 않을 수도 있다.

경고

쓰레드에 interrupt가 걸려 있으면 수행중인 쓰레드가 바로 중단되는 것이 아니라 interrupt status를 체크하여 Exception을 던져 더이상 진행하지 않도록 유도하는 것이기 때문에 발생하는 Exception에 대해서는 사용자 어플리케이션에서 처리해 주어야 한다. exception을 제대로 처리해 주지 않아서 발생하는 문제에 대해서는 사용자에게 책임이 있다.

또한 쓰레드에 interrupt가 걸린다고 해서 해당 쓰레드가 강제적으로 멈춘다든가, interrupt를 발생 시키는 순간에 바로 효과가 있는 것은 아니다. interrupt status를 체크하는 operation을 수행하려고 할 때 쓰레드가 interrupt에 걸려있으면 Exception이 발생하는 것이다.

쓰레드 interrupt에 대한 자세한 설명은 "JEUS Server 안내서 4.2.쓰레드 모니터링 및 제어" 장을 참고하기 바란다.

3. 표 설명
